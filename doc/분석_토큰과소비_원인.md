# LLM 토큰 과소비 원인 분석 (Python & venv 환경 중심)

**작성일**: 2026-04-15
**대상 프로젝트**: winresizer

## 1. 파이썬 및 `venv` 환경이 토큰 과소비의 원인이 될 수 있는가?

**결론부터 말씀드리면, "매우 그렇습니다."**
파이썬 가상 환경(`venv`) 자체가 문제라기보다는, LLM 에이전트(Gemini, Claude 등)가 해당 디렉토리를 처리하는 방식 때문에 심각한 토큰 낭비가 발생할 수 있습니다.

### 상세 원인
* **방대한 서드파티 라이브러리 코드**: `app/venv/` 디렉토리 내부에는 파이썬 표준 라이브러리뿐만 아니라, `Flask`, `pynput`, `pyobjc` 등 프로젝트에서 의존하는 수많은 외부 패키지의 소스 코드(`.py`)가 존재합니다.
* **에이전트의 무분별한 탐색**: 사용자가 "프로젝트 전체에서 특정 함수를 찾아줘"와 같은 포괄적인 명령을 내릴 때, 에이전트가 `venv` 폴더를 검색 대상에서 제외하지 않으면 수천 개의 무관한 라이브러리 파일을 스캔하거나 읽게 됩니다. 이때 발생하는 파일 경로, 검색 결과 텍스트, 코드 조각들이 모두 프롬프트 컨텍스트에 포함되어 막대한 토큰을 소비합니다.
* **현재 프로젝트의 방어 상태**: 다행히 현재 프로젝트의 `.gitignore` 파일에는 `app/venv/`와 `__pycache__/`가 등록되어 있습니다. 대부분의 최신 CLI 에이전트는 `.gitignore`를 존중하여 탐색 시 이를 제외하므로 기본적인 방어는 되어 있는 상태입니다. 하지만 에이전트 설정이나 특정 도구(예: 강제 전체 검색) 사용 방식에 따라 여전히 읽힐 위험이 존재합니다.

## 2. 프로젝트 구조상 예상되는 기타 토큰 과소비 원인

현재 디렉토리 구조와 프로젝트 특성을 분석했을 때, 다음과 같은 요소들이 토큰을 급격히 소모하게 만드는 주된 원인들입니다.

### A. 캐시 및 자동 생성 디렉토리 누락 (`.pytest_cache`)
* **현상**: 디렉토리 트리를 보면 루트와 `app/` 폴더 하위에 `.pytest_cache/` 디렉토리가 존재합니다. 하지만 `.gitignore`에는 이 폴더가 명시적으로 제외되어 있지 않습니다.
* **문제점**: 테스트 실행 결과와 캐시 데이터가 담긴 JSON 파일 등이 에이전트의 검색망에 걸려 불필요한 토큰을 낭비할 수 있습니다.

### B. 빌드 아티팩트 (PyInstaller)
* **현상**: `build.sh`와 `WinResizer.spec` 파일이 존재하며, 이는 앱을 빌드하여 `build/`, `dist/` 폴더를 생성함을 의미합니다.
* **문제점**: 이 폴더들은 `.gitignore`에 등록되어 있지만, 간혹 에이전트가 에러 디버깅 등을 이유로 빌드 로그나 결과물을 통째로 읽어오려 시도할 때 수만 줄의 바이너리/텍스트 혼합 데이터가 컨텍스트로 유입될 수 있습니다.

### C. 이미지 에셋 (`ref/` 폴더)
* **현상**: `ref/` 폴더 안에 여러 개의 `.png` 이미지 파일이 있습니다.
* **문제점**: 비전(Vision) 기능을 지원하는 모델의 경우, "UI 디자인을 분석해줘" 등의 요청 시 이미지 파일을 직접 컨텍스트에 로드(Base64 인코딩 등)할 수 있습니다. 이미지 1장은 수천~수만 토큰을 단일 턴에 소모하므로 주의가 필요합니다.

### D. 방대한 작업 이력 및 문서 (`doc/done/`, `doc/hold/`)
* **현상**: 이미 완료되었거나(`done/`) 보류 중인(`hold/`) 마크다운 문서가 매우 많습니다.
* **문제점**: 작업 시작 시 에이전트가 "컨텍스트 파악"을 명목으로 이 모든 문서들을 한 번에 읽도록 지시받는다면, 실제 코드 작성 전에 이미 컨텍스트 윈도우(Context Window)의 상당 부분을 과거 이력으로 채우게 되어 이후 대화의 매 턴마다 누적된 토큰 비용이 발생합니다.

### E. AGENTS.md의 'YOLO 모드 (연쇄 실행)' 특성
* **현상**: `AGENTS.md` 지침에 따라 에이전트는 사용자의 개입 없이 스스로 계획하고 여러 번의 도구 호출(Tool call)을 연쇄적으로 실행합니다.
* **문제점**: 에이전트가 잘못된 가설을 세우고 에러를 해결하기 위해 여러 번 코드를 수정하고 테스트(예: `pytest` 실행 -> 에러 로그 분석 -> 재수정)하는 과정을 10~20회 반복한다면, 그 모든 에러 로그와 시행착오 과정이 히스토리에 누적됩니다. 대화가 길어질수록 컨텍스트가 비대해져 마지막 턴에는 엄청난 양의 토큰이 한 번에 소비됩니다.

## 3. 해결 및 완화 방안 (가이드)

1. **Ignore 규칙 강화**: 
   * `.gitignore`에 `.pytest_cache/`를 반드시 추가하십시오.
   * 에이전트 전용 무시 파일(`.geminiignore` 또는 `.cursorignore` 등)을 만들어 `app/venv/`, `__pycache__/`, `.pytest_cache/`, `ref/`, `build/`, `dist/` 등을 명시적으로 차단하여 원천적으로 읽기를 금지하는 것이 좋습니다.
2. **검색(Grep) 및 읽기 범위 제한**: 에이전트에게 지시를 내릴 때, "앱 전체에서 찾아줘" 대신 "app/src/ 내부에서만 찾아줘"와 같이 경로를 좁혀주면 탐색 비용을 크게 줄일 수 있습니다.
3. **히스토리 정리 및 컨텍스트 초기화**: 에이전트가 연쇄 실행(YOLO) 중 막혀서 엉뚱한 로그만 잔뜩 생성했다면, 해당 세션을 계속 이어가기보다 대화를 초기화(New Chat)하고 새롭게 명확한 지시(Directive)를 내리는 것이 경제적입니다.
4. **문서 참조 최소화**: 에이전트에게 필요한 특정 스펙 문서(예: `spec/init_요구사항명세서.md`)만 명시적으로 읽도록 지시하고, 완료된 문서는 읽지 않도록 제한하십시오.
