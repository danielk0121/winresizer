# [버그] 작은 창 크기 제한 앱의 우측/하단 정렬 시 화면 짤림 및 사이클링 불가

## 1. 현상
- 사용자가 설정한 커스텀 비율(예: 35%)이 앱(크롬 등)의 최소 폭 제한보다 작을 경우:
  - **좌측 정렬**: 정상 작동하는 것처럼 보임. (x=0 기준이라 창이 오른쪽으로 더 넓어질 뿐 화면 안에는 있음)
  - **우측 정렬**: **창이 오른쪽 화면 밖으로 짤림.**
  - **모니터 이동**: 우측 정렬 상태에서 다음 모니터로 넘어가지 못함.

## 2. 원인 분석 (Root Cause)
### A. 수학적 계산과 물리적 제한의 충돌
- `CoordinateCalculator`는 앱의 최소 크기를 모른 채 `x = 모니터너비 * (1 - 0.35)`와 같이 계산함.
- 예: 모니터 너비 1000px, 설정 35% -> 계산된 x=650px, 너비=350px.
- 하지만 크롬의 최소 폭이 500px이라면, `set_window_bounds(x=650, w=500)`이 실행됨.
- 결과적으로 창의 오른쪽 끝은 **650 + 500 = 1150px**이 되어 모니터 경계(1000px)를 150px 초과하여 짤리게 됨.

### B. 사이클링 판정 로직의 한계
- 현재 엣지 판정 로직: `abs((현재x + 현재w) - (모니터x + 모니터w)) <= 20`.
- 위 예시에서 `1150 - 1000 = 150`이므로 오차가 너무 커서 "우측에 정렬됨"으로 인식하지 못함.
- 따라서 사용자가 아무리 단축키를 눌러도 로직은 "아직 우측 정렬이 안 되었으니 다시 정렬해"라고 판단하며 계속 같은 모니터에 머물게 됨.

## 3. 해결 방안 (Edge Case 대응 설계)
- **좌표 보정(Clamp) 로직 도입**:
  - 창 크기를 변경한 후, 혹은 변경하기 전에 해당 앱의 최소 크기를 고려해야 함.
  - 우측/하단 정렬 시에는 `x` 좌표를 `모니터우측경계 - 실제창너비`로 재계산하여 창이 왼쪽으로 확장되더라도 오른쪽 엣지는 반드시 모니터 경계에 붙도록 보정해야 함.
- **판정 로직 개선**:
  - 창이 짤린 상태(화면 밖 좌표)라도 해당 방향으로의 이동이 불가능한 상태라면 정렬된 것으로 간주하는 예외 처리 필요.

## 4. 구현 계획
- [x] `window_controller.py`에서 창 크기 적용 후 실제 적용된 크기를 다시 가져오는 로직 추가.
- [x] 우측/하단 정렬 시 실제 크기에 맞춰 `x`, `y` 좌표를 역으로 재계산(Re-anchoring)하는 보정 단계 추가.
- [x] 보정된 좌표를 기준으로 사이클링 판정 수행 (짤린 상태에서도 판정 가능하도록 개선).

## 5. 완료 보고 (2026-04-15)
- `execute_window_command` 함수에서 `set_window_bounds` 호출 직후 `actual_bounds`를 확인하여 짤림 발생 시 즉시 위치를 보정하도록 수정함.
- `is_already_aligned` 판정 로직에 `actual_right > expected_right + 5` 조건을 추가하여, 짤린 상태에서도 다음 모니터로의 사이클링이 정상 작동하도록 보장함.
- `app/tests/test_bug_window_overflow.py`를 통해 우측/하단 짤림 및 사이클링 동작을 검증함.
